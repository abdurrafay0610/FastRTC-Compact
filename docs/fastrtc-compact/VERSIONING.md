# Versioning Guide — fastrtc-compact

Scope: **repository files only.** Tagging, GitHub Releases, and PyPI publishing are
covered in the separate release guide.

---

## 1. Where the version number lives

### 1.1 `pyproject.toml` — the single source of truth (REQUIRED)

```toml
[project]
version = "0.0.1"   # <- change this
```

This is the **only place in the codebase where the version number is written**.
It is what pip resolves, what PyPI displays, and what the built wheel/sdist
filenames use (`fastrtc_compact-<version>-py3-none-any.whl`).

### 1.2 `CHANGES.md` — the changelog entry (REQUIRED by project convention)

Add a new section at the top for the release, replacing/closing out the
`[Unreleased]` section if one exists:

```markdown
## [0.0.2] — 2026-07-20

### Added
- ...

### Changed
- ...

### Fixed
- ...
```

Rule of thumb: the changelog entry and the `pyproject.toml` bump go in the
**same commit** — the one that gets tagged. This guarantees the tag, the
package version, and the changelog never drift apart.

### 1.3 Places that do NOT need a manual change (and why)

| Place | Why it's automatic |
| --- | --- |
| `backend/fastrtc/__init__.py` | No hardcoded `__version__` exists. If one is ever wanted, derive it — do not hardcode: `from importlib.metadata import version; __version__ = version("fastrtc-compact")` |
| Docs header version badge | Reads the latest **GitHub Release/tag** via the GitHub API at page load. Updates when the release is published (separate guide). |
| README / docs PyPI badge (`img.shields.io/pypi/v/fastrtc-compact`) | Rendered live from PyPI. Updates on its own after upload. |
| Install commands in README / `docs/index.md` | Written unpinned (`pip install fastrtc-compact`), so they never reference a specific version. Keep it that way. |
| Wheel / sdist filenames | Generated by the build from `pyproject.toml`. |

### 1.4 Sanity check before committing

```bash
# The only hardcoded version occurrences should be pyproject.toml and CHANGES.md:
grep -rn "0\.0\.1" --include="*.py" --include="*.toml" --include="*.md" \
  pyproject.toml backend/ README.md docs/index.md
```

(Replace `0.0.1` with the *old* version. If it shows up anywhere besides the
changelog history, something was hardcoded that shouldn't be.)

## 2. Version numbering convention (Semantic Versioning)

Format: **MAJOR.MINOR.PATCH** — e.g. `1.4.2`.

A small correction to the intuitive reading: the segments are defined by
**impact on users' code**, not by how much effort the change took.

| Segment | Bump when... | Examples |
| --- | --- | --- |
| **PATCH** (right) | Bug fixes and internal changes that are **fully backwards compatible**. A user upgrades and nothing in their code needs to change; things just work more correctly. | Teardown bug fix; a wrong resample ratio corrected; dependency pin adjusted; performance improvement with identical behavior. |
| **MINOR** (middle) | **New functionality, backwards compatible.** Existing code keeps working; new things become possible. Also: deprecations (old way still works but warns). | Streaming TTS/STT support added; `server_rtc_configuration` accepts a callable; new optional parameter on `Stream`. |
| **MAJOR** (left) | **Breaking changes.** Existing user code may stop working and requires migration. | Removing `additional_outputs_handler`; renaming the import from `fastrtc` to something else; changing `mount()` route paths. |

Two important refinements:

**Documentation and comment changes don't get a release at all.** If no line of
shipped code changed, there is nothing to publish — push to `main`, the docs
site redeploys, done. Only bump a version when the *installed package* differs.
(If a docs fix accidentally rides along in a code release, fine — it just
doesn't justify one by itself.)

**The 0.x exception (where this project currently lives).** SemVer officially
says that in `0.x.y`, anything may change at any time — there is no stability
promise yet. The widely used convention during 0.x is to shift everything
right by one:

| During 0.x | Meaning |
| --- | --- |
| `0.MINOR.x` bump (e.g. 0.0.x → 0.1.0) | New features **or** breaking changes |
| `0.x.PATCH` bump (e.g. 0.1.0 → 0.1.1) | Bug fixes only |

**`1.0.0` is a statement, not a milestone of size:** it says "the public API is
now stable; from here on, breaking changes cost a MAJOR bump." For
fastrtc-compact a sensible bar for 1.0.0 would be: the huggingface_hub/vendored
VAD work done, streaming STT/TTS/LLM support landed, and the API surface you
intend to commit to (exports in `__init__.py`) audited and final.

### Quick decision test

Ask: *"If someone pinned the previous version and upgrades, can their code break?"*

- Yes → MAJOR (or 0.MINOR during 0.x)
- No, but they can now do something new → MINOR
- No, and nothing new — things are just fixed/faster/smaller → PATCH

### Pre-releases (optional, for later)

When a risky change needs testing before the real release, PEP 440 pre-release
suffixes keep the version number unburned: `0.2.0rc1`, `0.2.0b1`. pip ignores
these unless explicitly requested (`pip install --pre`), so normal users never
get them by accident.

## 3. The complete in-repo checklist

1. [ ] Decide the new version using §2 (breaking / feature / fix?)
2. [ ] Edit `version = "..."` in `pyproject.toml`
3. [ ] Add the `CHANGES.md` entry (dated, categorized)
4. [ ] Run the sanity grep from §1.4
5. [ ] Commit both files together: `git commit -m "Release 0.0.2"`

Everything after this commit (tag, GitHub Release, build, PyPI upload) is in
the release guide.
